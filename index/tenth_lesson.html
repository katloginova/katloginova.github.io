GIT и GitHub (урок 10)
GIT – это одна из систем контроля версий. Это значит, что она хранит всю историю изменений проекта. История Вашего проекта и история изменений этого
Первым делом мы загрузим Git. Для пользователей Windows необходимо установить Git Bash. После завершения установки приступим к регистрации аккаунта GitHub. Итак, у нас теперь есть Git, инструмент командной строки, и GitHub аккаунт, куда мы будем загружать свои репозитории.
Используя Git Bash перейдем в корневую директорию проекта. Для начала проверим в какой папке мы находимся с помощью команды «pwd»
Затем при необходимости перейдем в нужную папки с помощью команды:
cd /путь папки (можно мышкой перетащить саму папку и путь пропишется сам)
Первым делом необходимо установить Git для данной папки проекта. Делается это с помощью команды (создаст «.git» репозиторий в проекте.):
git init
Затем пропишем две команды, которые определят информацию, которая будет использоваться при каждом commit (фиксирование изменений). Их стоит выполнить всего один раз при первичной установке Git:
git config --global user.name "Ваше Имя"
git config --global user.email "ВашаПочта@mail.com"
Команда, которая создает файл с названием «.gitignore». В нем необходимо прописать названия файлов или директорий, изменения в которых не нужно отслеживать (они будут игнорироваться Git). Изменения в игнорируемых файлах не будут отображаться при выполнении git status (этот файл можно создать также вручную без команды):
touch .gitignore
Следующая команда добавит файл в “staging area”(участок подготовки). staging area – это своего рода секция, в которой файлы проходят подготовку к перемещению в наш репозиторий:
git add имяФайла.расширение
Если необходимо добавить всё из директории проекта в staging area:
git add .
Если необходимо добавить все файлы с расширением .html в staging area (расширение можно менять):
git add *.html
Чтобы показать, что уже было добавлено в staging area и какие файлы были изменены и ждут перемещения в staging area:
git status
Если необходимо убирать выбранный файл из staging area:
git reset имяФайла.расширение
Команда для создания коммита, которая берёт файлы из staging area и “фиксирует” их в локальный репозиторий. В кавычки следует вставить краткое описание изменений для конкретного коммита:
git commit -m "описание коммита"
Показывает список коммитов, которые у нас есть в проекте, команда:
git log
Если нужно сравнить текущее состояние проекта с указанным коммитом (id_commit), то:
git diff id_commit
Команда для создания ветки:
git branch add названиеВетки
Переключает контроль над созданной веткой и работает в её пределах:
git checkout названиеВетки
Создает ветку и переключается на нее:
git checkout -b названиеВетки
Находясь в какой-то из веток, эта команда берет коммиты из ветки, которая указана и добавляет в ту на которой мы находимся:
git merge названиеВетки
При выполнении команды «merge» может произойти конфликт, который происходит, если в одном файле совпадают строки. В таком случае мы исправляем все необходимой в файле и создаем коммит без описания. И тогда используется описание предложенное самим Git.

Как подключить проект на удаленный репозиторий?

Перед тем, как подключить наш проект на удаленный репозиторий, необходимо для начала добавить ключ из локального компьютера на GitHub. Открываем утилит Git GUI, а там на вкладке Help выбираем Show SSH key и генерируем ключ. Затем этот ключ копируется и вставляется на GitHub. Ключ можно также сгенерировать с помощью команд в консоли. Для этого перейдем в папку «ssh» с помощью команды «cd ~/ssh». Далее с помощью команды «ssh-keygen» генерирую ключ, а с помощью команды «cat id_rsa.pub» – вывожу ключ в консоль.

Для того, чтобы подключить полученный ключ на GitHub, заходим в Setting –> SSH and GPG keys –> New SSH key и добавляем скопированный ключ.

Теперь привяжем наш проект с компьютера на удаленный репозиторий с помощью команды:
git remote add origin git@github.com:katloginova/progect_hillel.git
где git@github.com:katloginova/progect_hillel.git – путь SSH на наш удаленный репозиторий
После этого можно загружать наш проект на репозиторий с помощью команды:
git push origin имяВетки(master)
GIT, GitHub и GitHub Pages (урок 11)
Команда, которая показывает какие есть ветки в проекте:
git branch
Команда для того, чтобы загрузить проект с удаленного репозитория на компьютер, где нет данного проекта (название папки соответствует названию репозитория на GitHub):
git clone git@github.com:katloginova/progect_hillel.git
По умолчанию из удаленного репозитория затягивается только одна ветка «master». Команда «git fetch» связывается с удалённым репозиторием и забирает из него все изменения, которых у нас пока нет и сохраняет их локально. Для того, чтобы затянуть остальные ветки выполняется команда:
git fetch
Команда «git pull» работает как комбинация команд «git fetch» и «git merge», т.е. Git вначале забирает изменения из указанного удалённого репозитория, а затем пытается слить их с текущей веткой.
git pull origin master
Команда, которая отменяет процесс слияния веток:
git merge --abort
Для того, чтобы вернуться к какому-то коммиту есть команда «git reset id_commit» (можно использовать вместо «id_commit» «head», так называется последний коммит). Есть два режима «reset»:
  • обычный «git reset id_commit» удаляет коммиты, но не изменяет сами файлы
  • жесткий «git reset id_commit --hard» удаляет не только коммиты, а сбрасывает все файлы до состояния последнего оставшегося коммита
Но так делать можно только, если эти коммиты, которые мы удаляем еще не за-push на удаленный репозиторий, иначе будет конфликт. Обычно на практике «reset» используется только для удаления последнего коммита. Есть альтернативная команда, которая не удаляет коммиты, а делает новые коммиты такими, как будто мы вернулись к тому состоянию проекта, коммит которого мы указываем вместе с этой командой:
git revert commit
Если необходимо удалить привязку к удаленному репозиторию используется команда:
git remote rm origin

Как выложить наш проект на GitHub Pages?

GitHub Pages – это бесплатный хостинг для статических файлов, но размещать можно только один проект.
Для того, чтобы выложить наш проект на GitHub Pages зайдем на GitHub в наш проект и на вкладке «Settings» переименуем проект на следующее имя:
имя_пользователя.github.io (katloginova.github.io)
Если пролистать ниже содержимое вкладки «Settings», то можно увидеть адрес-ссылка, по которой можно перейти на страницу проекта:
Your site is published at https://katloginova.github.io/

Пошаговое использование Git с нуля проекта

  1. Когда только создали папку с проектом еще до начала работы с самим проектом, необходимо создать пустой локальный репозиторий «.git» с помощью «git init».
  2. Создаем файл «.gitignore» в папке проекта, где указываем все файлы или папки, за которыми Git не нужно следить:
    • папки записываются – «/имя_папки»;
    • файлы записываются – «имя_файла».
  3. Следующим шагом будет создание начального коммита, желательно с комментарием «initial» с помощью «git add .» и «git commit -m”project initial”»
  4. После этого сразу создадим и привяжем репозиторий на GitHub с помощью «git remote add origin путь(ssh)» и «git push origin master».
